# 🛡️ Инфраструктурное усиление и защита (Hardening)

В ходе эксплуатации сервера была выявлена массированная атака ботов (сканеры SOCKS5 и Telegram-прокси), которая приводила к исчерпанию ресурсов CPU (200-300%) и падению сайта. Данный документ описывает внедренные меры защиты.

## 1. 🛡️ Сетевой экран Hetzner Firewall (Самое важное)

Для предотвращения нагрузки на CPU, бот-трафик должен отсекаться ДО того, как он попадет на сервер.

### Текущая настройка:
Порты `80` (HTTP) и `443` (HTTPS) разрешены **ТОЛЬКО** для IP-адресов Cloudflare. Все остальные прямые подключения к IP сервера на эти порты блокируются сетью Hetzner.

### Список IP Cloudflare:
Актуальный список всегда доступен по адресу: https://www.cloudflare.com/ips/

### Как обновить правила через API:
Если Cloudflare обновит свои IP, их нужно добавить в Firewall `firewall-quiz-hub` (ID: `10820247`).

## 2. 🚫 Fail2ban (Автоблокировка ботов)

Если какой-то бот все же просочится или атакует через Cloudflare, Fail2ban заблокирует его на уровне `iptables`.

- **Логи**: Nginx пишет логи в `./logs/nginx/access.log` (монтируется на хост).
- **Фильтр**: `/etc/fail2ban/filter.d/nginx-bots.conf` (ищет попытки проксирования и SOCKS5).
- **Действие**: Бан на 24 часа после 10 подозрительных запросов за 30 секунд.

**Управление:**
```bash
# Проверить статус
sudo fail2ban-client status nginx-bots

# Разбанить IP
sudo fail2ban-client set nginx-bots unbanip <IP-ADDRESS>
```

## 3. ⚙️ Оптимизации Nginx (`nginx-prod.conf`)

Конфигурация Nginx была усилена для борьбы с "медленными" атаками и сканерами:

- **`worker_connections 4096`**: Увеличено количество одновременных соединений.
- **`ssl_reject_handshake on`**: (в дефолтном блоке) Сбрасывает SSL-соединения, если запрос идет не на ваш домен, экономя CPU на расшифровке.
- **Таймауты**: Ограничены до 10с (`keepalive_timeout`, `client_body_timeout` и т.д.), чтобы боты не занимали слоты соединений.
- **`limit_conn` / `limit_req`**: Лимиты 20 соединений на IP и 30 запросов в секунду.

## 4. 🌐 Исправление DNS и Docker

Для корректной работы `docker compose build` на сервере были применены следующие системные настройки:

- **/etc/resolv.conf**: Установлены публичные DNS (8.8.8.8, 1.1.1.1).
- **/etc/docker/daemon.json**:
  ```json
  {"dns": ["8.8.8.8", "1.1.1.1"]}
  ```
  Это позволяет контейнерам успешно скачивать пакеты во время сборки.

## 📋 Чеклист при миграции на новый сервер

1. [ ] Обновить DNS в `/etc/resolv.conf` и `/etc/docker/daemon.json`.
2. [ ] В панели Hetzner создать Firewall и разрешить 80/443 **только** для Cloudflare IP.
3. [ ] Склонировать репозиторий и запустить `./start-prod.sh`.
4. [ ] Настроить Fail2ban скриптом: `chmod +x scripts/setup-fail2ban.sh && scripts/setup-fail2ban.sh`.
