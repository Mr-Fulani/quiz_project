# Переход global chat с polling на WebSocket

## Цель
Перевести общий чат на постоянное соединение для уменьшения задержек и нагрузки от частых HTTP-запросов.

## Что уже подготовлено в итерации 1
- Выделены отдельные модели `GlobalChatMessage` и `GlobalChatBan`.
- API и UI разделены по маршрутам `chat/*`, что упростит замену слоя доставки событий.
- Вся бизнес-логика бана и модерации находится в backend и может быть переиспользована в websocket-consumer.

## План второй итерации
1. Подключить `django-channels` и channel layer на Redis.
2. Добавить consumer для комнаты общего чата в рамках текущего tenant.
3. Перенести отправку/получение сообщений на websocket-события, сохранив HTTP-endpoint как fallback.
4. Передавать события модерации (удаление, бан/разбан) через websocket broadcast.
5. Обновить фронтенд: fallback на polling при недоступном websocket.

## Критерии готовности
- Новые сообщения появляются без polling.
- Действия модератора видны всем участникам сразу.
- При временной недоступности websocket чат продолжает работать через HTTP fallback.
